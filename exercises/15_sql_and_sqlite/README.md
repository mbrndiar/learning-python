### Connection and relational operations

- Configure each connection with `sqlite3.Row` and enforced foreign keys.
- Create constrained `projects`, `tasks`, and affinity-demo tables with a
  multi-statement schema script, including integer-only priorities from 1
  through 5 despite SQLite's permissive type affinity.
- Insert one project or task and return the ID generated by the individual
  `execute()` cursor.
- Batch tasks with `executemany()`.
- Map rows into `Task` values and implement filtered, ordered CRUD, including
  the `False` path for a missing update or delete. Result limits must be
  non-negative; zero returns no rows and a negative limit raises `ValueError`.

### Queries and SQLite behavior

- Join task titles to project names.
- Summarize every project with `LEFT JOIN`, `COUNT`, `GROUP BY`, and a
  parameterized `HAVING` threshold.
- Add a lookup index and return non-empty `EXPLAIN QUERY PLAN` details without
  depending on exact version-specific wording.
- Explicitly begin multi-write transactions so failed batches and task pairs
  roll back completely even on a connection using `isolation_level=None`.
- Demonstrate SQLite affinity by returning each stored value's `typeof(...)`
  result and Python value.

### One contract, two adapters

- Implement `InMemoryTaskRepository`.
- Implement `SQLiteTaskRepository` over an injected, caller-owned connection.
- Complete the highest-priority open task through the `TaskRepository`
  protocol.

The supplied `assert_repository_contract()` runs the same observable behavior
against both fresh adapters. It deliberately does not inspect private storage,
SQL text, or performance.
